# 两个 Plus 账号还不够省心：我做了一个 Codex 多账号用量工具

![文章封面](./wechat-cover.png)

> 作者：@可以叫我才哥  
> 项目：Codex Usage Float  
> 关键词：Codex、AI 编程、多账号管理、Token 统计、Electron

最近一段时间，我在日常开发中使用 Codex 的频率越来越高。

从阅读陌生项目、排查问题，到实现功能、补测试、整理文档，再到打包发布，很多过去需要来回切换工具的工作，现在都可以在一个连续的对话里完成。它不是简单地“帮我补几行代码”，而是越来越像一个可以一起读代码、做判断、执行和验证的工程搭档。

但使用强度上来以后，一个很现实的问题也出现了：**一个 Plus 账号的使用额度，经常覆盖不了我完整的日常开发需求。**

于是，我购买了两个 Plus 账号。

额度问题暂时缓解了，新的麻烦却随之而来：我需要反复确认当前登录的是哪个账号、哪个账号还有额度、5 小时窗口什么时候重置、周额度还剩多少；切换账号时，还要退出、登录、重新确认状态。有时刚切完账号，正在运行的 Codex 会话仍然保留旧的认证状态，还得重启才能恢复正常。

这些操作单次看起来都不复杂，但一天重复很多次，就会不断打断工作节奏。

于是我开始想：

**能不能做一个小工具，把多个账号的用量放在同一个面板里，同时提供明确、克制、可确认的手动账号切换？**

这就是 Codex Usage Float 的起点。

---

## 一、先简单聊聊 Codex 是什么

如果你还没有使用过 Codex，可以先把它理解成一个能够直接进入项目、读取文件、修改代码、执行命令并验证结果的 AI 编程助手。

传统的 AI 编程使用方式，往往是把一段代码复制到聊天窗口里，再把回答复制回编辑器。Codex 更接近真实的软件开发流程：它可以在你的项目目录中工作，理解现有结构，遵循仓库约定，修改文件，运行测试，再把结果交给你审查。

它比较适合处理这些任务：

- 阅读陌生代码，梳理模块关系和调用链；
- 根据明确需求实现功能；
- 定位错误、分析日志并修复问题；
- 为现有功能补充测试；
- 调整前端 UI，并通过截图或浏览器检查结果；
- 编写 README、变更日志和发布说明；
- 执行 Git 提交、打包、构建等工程流程。

OpenAI 提供了 Codex CLI、桌面应用、IDE 扩展等不同使用方式。本文主要围绕本地 Codex 的使用体验展开。

### 1. 安装 Codex CLI

如果本机已经安装 Node.js，可以使用 npm 安装：

```powershell
npm install -g @openai/codex
```

安装完成后，在终端中进入项目目录：

```powershell
cd D:\your-project
codex
```

首次启动时，根据界面提示使用 ChatGPT 账号或 API Key 完成官方登录。

官方资料：

- [Codex 快速开始](https://developers.openai.com/codex/quickstart)
- [Codex CLI 文档](https://developers.openai.com/codex/cli)
- [Codex 认证说明](https://developers.openai.com/codex/auth)
- [OpenAI Codex 官方仓库](https://github.com/openai/codex)

### 2. 第一次使用，不要急着让它“大改”

我更推荐从一个边界清楚的小任务开始。例如：

```text
请先阅读项目结构和 README，不要修改代码。
帮我说明这个项目如何启动、核心入口在哪里、现有测试怎么运行。
```

当它完成项目理解后，再给出具体任务：

```text
请修复设置弹窗在窄窗口下按钮重叠的问题。
保持现有设计风格，修改后运行语法检查，并截图验证。
```

这种方式比一句“帮我优化项目”更有效。一个好任务通常包含四个部分：

1. **目标**：希望解决什么问题；
2. **范围**：允许修改哪些模块；
3. **约束**：哪些行为、样式或数据不能改变；
4. **验收**：需要运行哪些检查，最终怎么看算完成。

### 3. 让 Codex 先读项目，再做判断

我在实际使用中非常看重这一点。即使需求看起来简单，也应该让它先理解代码库，而不是凭空设计一套新结构。

如果项目里有 `AGENTS.md`，还可以写入团队或个人约定，例如：

```markdown
# 开发约定

- 使用中文沟通。
- 修改前先阅读现有实现。
- 不要覆盖无关改动。
- 前端改动后需要截图验证。
- 未经明确要求，不要自动推送或部署。
```

这类规则可以减少重复说明，也能让每次协作更稳定。

### 4. 把“验证”也交给 Codex，但不要放弃审查

AI 写完代码不等于任务完成。一个完整流程至少应该包括：

- 查看实际修改内容；
- 运行语法检查或测试；
- 启动程序验证关键流程；
- 前端任务检查文字溢出、按钮重叠和交互状态；
- 最后再决定是否提交和发布。

Codex 可以执行这些步骤，但最终的产品判断仍然应该由人来做。尤其涉及认证文件、账号切换、删除数据和线上部署时，更要清楚每一步会产生什么后果。

## 二、为什么我需要两个 Plus 账号

我的使用场景不是偶尔问几个代码问题，而是把 Codex 放进连续的开发流程中。

一个功能从需求到完成，可能会经历：

```text
分析项目
→ 调研实现方式
→ 修改主进程和前端
→ 启动程序
→ 查看截图
→ 根据截图继续调整
→ 运行测试
→ 更新 README
→ 打包 EXE
→ 验证最终产物
```

当任务较复杂、上下文较长，或者需要频繁读取文件和反复验证时，用量会增长得很快。一个 Plus 账号有时会在关键阶段进入额度等待窗口。

为了不让开发节奏被中断，我购买了第二个 Plus 账号。这两个账号都是我自己正常购买和使用的账号，我的目标也不是绕过平台限制，而是更清楚地管理自己已有账号的状态。

真正让我觉得麻烦的，是下面这些细小但高频的问题：

- 当前 Codex 到底登录了哪个账号？
- 这个账号的 5 小时额度还剩多少？
- 1 周额度还剩多少？
- 下一次重置是什么时候？
- 有没有可用重置卡？
- 另一个账号当前是什么状态？
- 最近本地所有会话到底消耗了多少 Token？
- 切换账号后，为什么当前会话有时仍然不能使用？

如果每次都打开网页、退出登录、重新登录，再回到 Codex 检查，注意力会被不断切碎。

**一个工具真正有价值的地方，往往不是替你完成某个惊天动地的大功能，而是消除这些每天都会发生的小摩擦。**

## 三、从痛点到需求：我最初只想看用量

最初的想法很简单：做一个 Windows 桌面悬浮球，常驻显示当前账号 5 小时窗口的剩余百分比，点开后查看详细用量。

第一版需求只有这些：

- 5 小时剩余百分比与重置时间；
- 1 周剩余百分比与重置时间；
- 当前会员等级；
- 可用重置卡；
- 一个轻量的桌面悬浮入口。

但真正开始使用后，我很快发现“只显示当前账号”并不能解决根本问题。既然我有两个账号，我真正需要的是**同时看到多个账号的状态**，而不是切过去以后才知道它有没有额度。

于是产品形态从“单账号用量浮窗”逐渐变成了“多账号用量看板”。

![多账号用量看板](../screenshots/multi-account-dashboard.png)

现在，每个账号卡片会同时展示：

- 个人资料中的显示昵称和用户名；
- Plus、Pro 等会员等级；
- 会员到期时间；
- 5 小时窗口剩余量和重置时间；
- 1 周窗口剩余量和重置时间；
- 今日、7 天和 30 天的账号 Token 概览；
- 当前使用状态与手动切换入口。

为了让额度状态更容易扫一眼判断，我采用了红、橙、黄、绿四档颜色：

| 剩余量 | 颜色 | 含义 |
| --- | --- | --- |
| 0% - 10% | 红色 | 接近耗尽 |
| 11% - 25% | 橙色 | 需要留意 |
| 26% - 50% | 黄色 | 中等余量 |
| 51% - 100% | 绿色 | 余量充足 |

百分比显示的是“剩余量”，而不是“已使用量”。这是一个小细节，但对于日常决策更直接：看到 8%，我马上知道应该停止在这个账号上继续启动大任务。

## 四、调研阶段：数据到底从哪里来

UI 不是这个工具最难的部分。真正困难的是：**哪些数据真实存在，哪些数据能稳定获取，哪些数据看起来能拆分、实际上却不能可靠拆分。**

我先阅读了 OpenAI 官方 Codex 文档和官方仓库，再参考了一些社区开源项目的思路：

- [170-carry/codex-tools](https://github.com/170-carry/codex-tools)
- [lyssl/codex_usage](https://github.com/lyssl/codex_usage)
- [OpenAI Codex 官方文档](https://developers.openai.com/codex)
- [openai/codex](https://github.com/openai/codex)

调研之后，我把数据分成了三类。

### 1. 当前认证信息

本地 Codex 登录后，会在 `~/.codex/auth.json` 中保存认证状态。这个文件可以帮助工具判断当前账号，并在授权范围内请求账号相关信息。

它也是多账号切换的关键，但同时是整个工具最敏感的部分，因为其中包含登录令牌。

### 2. 账号级用量与资料

账号接口可以返回或推导：

- 显示昵称、用户名；
- 会员等级与到期时间；
- 5 小时、1 周用量窗口；
- 重置时间；
- 可用重置卡；
- 部分账号级 Token 汇总。

这些数据适合展示账号当前状态，但接口字段可能变化，返回也可能有延迟。因此工具不能把一次空响应直接当成“账号没有数据”，而应该保留最近一次成功快照。

### 3. 本地会话日志

Codex 本地会话保存在：

```text
~/.codex/sessions
~/.codex/archived_sessions
```

其中的 JSONL 日志包含 `token_count` 事件，可以看到更细的 Token 分类：

- 输入 Token；
- 缓存输入 Token；
- 输出 Token；
- 推理输出 Token；
- 总 Token。

这部分数据非常有价值，但也有一个必须承认的限制：**多个账号共用同一个 `~/.codex` 时，历史会话日志通常没有足够稳定的账号标识，无法保证精确拆分归属。**

我曾认真研究过按账号 ID、会话时间、认证快照和指纹进行归属，但这些方法都可能在“同一个项目、同一个会话、期间切换账号”的场景中产生误差。

最终，我选择了一个更诚实的产品口径：

> 账号卡片展示账号级概览；底部本地日志汇总统计所有会话，并明确标注“不拆分账号”。

比起给用户一个看起来很精确、实际上可能归属错误的数字，我更愿意把能力边界直接写在界面上。

## 五、账号切换：能替换文件，不代表当前进程立刻生效

多账号切换的基本思路是：

1. 用户通过官方流程登录账号 A；
2. 工具导入当前 `auth.json`，保存为账号 A 的本地快照；
3. 用户通过官方流程登录账号 B；
4. 再次导入，保存账号 B 的快照；
5. 后续在工具中手动选择账号，确认后替换当前 `auth.json`。

切换前必须弹出确认框，不做自动轮换，也不做“额度用完自动切下一个账号”。

![账号切换确认](../screenshots/account-switch-confirm.png)

写到这里时，我遇到了一个很容易误判的问题：文件已经成功替换，但正在运行的 Codex 仍然不能正常使用新账号。

经过验证，切换逻辑在文件层面是真实有效的，目标账号快照确实写入了 `auth.json`。问题在于已经运行的 Codex 进程可能把认证状态缓存在内存中。

这意味着：

- 替换 `auth.json` 不等于刷新当前进程的登录状态；
- 切换后通常需要完全关闭并重新启动 Codex；
- 如果重启后仍不可用，应通过官方界面重新登录；
- 工具不能宣称自己完整模拟了官方登录流程。

因此，我在界面中明确写了“手动切换，仅替换 auth.json”，而不是包装成无感、自动的账号池。

这种克制其实很重要。工具应该降低操作成本，但不能掩盖底层行为和风险。

## 六、本地 Token 汇总：从累计值里算出真实增量

本地日志里的 `token_count` 并不是每一条都可以直接相加。有些事件记录的是会话累计值，如果粗暴求和，会造成严重重复计算。

工具需要按会话读取事件，用当前累计值减去上一个累计值，得到本次新增 Token，再分别归入今日、7 天和 30 天窗口。

同时还要处理：

- 重复事件；
- 无效 JSON 行；
- 已归档会话；
- 累计值回退或新会话重新开始；
- 输入为 0 时缓存率不能除零；
- 推理输出是否已经包含在输出口径中。

最终的汇总区域会展示输入、缓存输入、缓存率、输出、推理输出、总计、文件数和事件数。

后来，我又加入了 GPT-5.5 标准 API 费用估算：输入费用同时考虑非缓存输入和缓存输入，输出费用单独计算，并把合计放在标题旁边。

![本地 Token 与费用汇总](../screenshots/local-token-summary.png)

这里要特别说明：这个金额只是把本地 Token 按标准 API 单价换算成一个直观参考，**不是 Plus 订阅账单，也不是实际扣费金额**。长上下文、Fast、Priority、Batch、Flex 等费率差异也无法仅凭汇总日志精确还原。

## 七、UI 是在一次次真实使用中磨出来的

这个工具的 UI 并不是一次设计完成的。

最初的面板只显示一个账号，后来逐步经历了这些调整：

- 从单账号变成多账号同时展示；
- 显示个人资料中的昵称与用户名，而不是令牌里的模糊字段；
- 当前账号增加明显但不过度的绿色状态；
- 会员等级和到期时间移动到账号标题同一行；
- 5 小时和 1 周区域统一布局；
- 重置时间放到剩余百分比后面；
- 重置卡不再每张套一个独立卡片，而是在一个列表中排列；
- 重置卡超过两张时，仅列表内部滚动；
- 本地日志固定放在底部；
- 今日、7 天、30 天切换后重新测量面板高度；
- 修复底部按钮上移、压住统计文字的问题；
- 输入和输出费用直接跟在 Token 值后面，减少额外占用高度。

我在开发过程中反复做的一件事，就是启动真实 Electron 应用、截图、检查文字是否溢出，再继续调整。

有一个典型问题是：连续切换“今日、7 天、30 天”后，内容高度发生变化，底部“打开网页”和“退出”按钮会上移，压到上一行统计文字。单看 CSS 很容易忽略，但在真实交互中非常明显。

最后的处理不是简单增加一个固定高度，而是让面板在渲染完成后重新测量内容，并为统计行与底部操作区保留稳定间距。这样既能适应不同账号数量，也能适应重置卡和日志数据变化。

这段经历让我再次确认：

> UI 的完成标准不是“代码看起来合理”，而是用户连续操作以后，界面仍然稳定。

## 八、技术实现：一个不大的 Electron 应用

项目使用 Electron 开发，整体结构并不复杂：

```text
src/main.js              主进程、窗口、数据同步、账号库
src/preload.js           安全 IPC 桥接
src/renderer/index.html  面板结构
src/renderer/app.js      渲染与交互
src/renderer/styles.css  UI 样式
```

主进程负责：

- 读取本地认证与状态文件；
- 请求账号用量和资料；
- 扫描本地会话日志；
- 管理多账号快照；
- 原子替换 `auth.json`；
- 调整悬浮球和详情面板窗口尺寸。

渲染进程只通过 preload 暴露的受控 IPC 与主进程通信，不直接获得 Node.js 的完整文件访问能力。

账号快照保存在：

```text
%APPDATA%/codex-usage-float/accounts.json
```

切换时会先写临时文件，再替换正式 `auth.json`，尽量降低写入中断造成文件损坏的风险。

Windows 版本使用 `electron-builder` 打包为 portable EXE。每次发布前，我都会检查：

- 主进程、preload、renderer 脚本语法；
- `app.asar` 是否包含最新源码；
- EXE 的修改时间、大小和 SHA-256；
- 实际启动的是打包后的 EXE，而不是开发命令；
- 面板是否能读取账号；
- 三个日志周期是否能切换；
- 底部按钮是否重叠；
- 测试进程是否完整退出。

当前版本为 `1.0.1`，作者信息为 `@可以叫我才哥`。

## 九、现在这个工具能做什么

目前 Codex Usage Float 已经具备以下能力：

1. 桌面悬浮球显示当前账号 5 小时剩余量；
2. 同时展示多个账号的 5 小时与 1 周额度；
3. 展示昵称、用户名、会员等级和到期时间；
4. 展示当前账号可用重置卡；
5. 手动导入当前账号；
6. 通过确认框手动切换本地账号快照；
7. 展示账号今日、7 天、30 天 Token 概览；
8. 汇总所有本地会话的输入、缓存输入和输出；
9. 按时间范围估算 GPT-5.5 标准 API 费用；
10. 打包为 Windows portable EXE。

## 十、如何导入第二个账号

如果你和我一样有多个自己正常购买的账号，可以按下面的方式操作：

1. 先通过 Codex 官方登录流程登录账号 A；
2. 打开工具，点击“导入当前账号”；
3. 在 Codex 中退出账号 A，并通过官方流程登录账号 B；
4. 回到工具，再次点击“导入当前账号”；
5. 两个账号会同时出现在账号看板中；
6. 点击非当前账号的“切换”；
7. 确认替换 `auth.json`；
8. 完全关闭并重新启动 Codex。

请不要把它理解成自动养号、自动轮换或共享账号工具。它只是把你本机已有的认证快照整理起来，并在你确认后手动切换。

## 十一、开发这个工具之后，我的一些思考

### 1. AI 编程最有价值的地方，是缩短“想法到验证”的距离

这个工具从一个很具体的个人痛点开始：我不想每天反复打开网页确认额度。

过去，这样一个小工具可能因为涉及 Electron、接口调研、日志解析、UI、Windows 打包而被一拖再拖。现在借助 Codex，我可以快速搭起第一版，再把更多时间放到真正需要人判断的地方：数据是否可信、交互是否克制、风险是否讲清楚。

### 2. “能实现”不等于“应该实现”

开发过程中，我考虑过自动切换账号、账号 Token 精确拆分、用量耗尽后自动轮换等功能。

但越深入研究，越发现这些功能可能带来错误归属、认证风险和不可控行为。最终我主动放弃了自动轮换，也没有把不可靠的本地日志强行拆到每个账号。

一个工具的成熟，不只体现在增加了多少功能，也体现在知道哪些功能暂时不该做。

### 3. 面对不稳定数据，诚实比“看起来完整”更重要

账号接口可能变化，Token 可能延迟，重置卡有效期也可能只能推算。

因此工具会保留最近一次成功快照，标记数据存在延迟，并在 README 中写清数据口径。用户看到的每一个数字，都应该知道它来自哪里、可能有什么误差。

### 4. 小工具首先应该解决自己的真实问题

我没有一开始就设计复杂的商业模式，也没有把它包装成庞大的“AI 开发平台”。它首先是一个我每天愿意打开、能减少几次打断的小工具。

如果它也恰好解决了你的问题，那就是开源带来的额外价值。

## 十二、最后的安全提醒

这个工具涉及本地 Codex 认证快照，使用前请理解以下事项：

- `accounts.json` 中包含敏感认证信息，不要上传或分享；
- 当前版本没有对账号快照额外加密；
- 只建议在可信的个人 Windows 设备上使用；
- 不要高频切换、自动轮换或进行异常请求；
- 切换失败时应回到官方登录流程；
- 用量、重置时间和费用估算仅供个人参考；
- 项目不是 OpenAI 官方产品，也不受 OpenAI 背书；
- portable EXE 当前未进行商业代码签名，Windows 可能提示未知发布者。

项目按照 MIT License 开源。它不出售账号、额度、会员或代充服务，也没有付费解锁和广告 SDK。

## 结语

我购买两个 Plus 账号，是为了让高频开发不被额度等待打断；我做这个工具，是为了让账号和用量管理不再反过来打断开发。

从一个“能不能看一下剩余额度”的小想法，到多账号看板、Token 汇总、费用估算、账号切换和 EXE 打包，这个项目本身也成了一次很完整的 AI 协作开发实践。

我越来越觉得，AI 编程真正改变的不是“写代码更快”这么简单，而是让很多过去因为成本太高而被搁置的小想法，终于有机会被认真做完。
